## 📋 当前状态

- ✅ **APP端签名**: 已完整实现，自动为所有API请求添加签名头
- ✅ **API端核心**: 签名验证算法、Nonce管理、时间戳检查已完成

## 🎯 一键启用全面签名验证

### **步骤 1: 激活签名中间件**

编辑 `record-api/src/main.rs`，取消注释并启用中间件：

```rust
// 第98行 - 启用签名验证保护
.layer(axum::middleware::from_fn_with_state(signature_state.clone(), signature_middleware))
```

### **步骤 2: 编译并重启API服务器**

```bash
cd record-api
cargo build --release
./target/release/record-api
```

### **步骤 3: 验证签名生效**

此时：
- ✅ 没有签名的请求会被拒绝（400错误）
- ✅ 过期签名会被拒绝（401错误）
- ✅ 重复Nonce会被拒绝（403错误）
- ✅ APP端会自动处理所有签名

## 🔍 验证签名是否生效

### **方法 1: 使用curl测试**

```bash
# 无签名的请求（应该被拒绝）
curl -v http://localhost:3001/api/profile
# 预期: 400 - Missing required signature headers

# 正确签名的请求需要所有4个头信息
curl -H "X-App-Key: record_app_v2" \\
     -H "X-Timestamp: 1625097600" \\
     -H "X-Nonce: abc123def456" \\
     -H "X-Signature: your_signature_here" \\
     http://localhost:3001/api/profile
```

### **方法 2: 使用APP测试**

APP会自动处理签名，所以：
- 如果APP能正常调用API，说明签名工作正常
- 如果APP收到401/403错误，说明签名验证失败
- 如果API服务器报错，说明中间件配置问题

## 🛡️ 安全级别说明

### **启用后提供的保护**

1. **防数据篡改**
   - 任何请求数据修改都会使签名无效
   - 攻击者无法伪造有效签名

2. **防重放攻击**  
   - Nonce确保每个请求只能用一次
   - 时间戳确保请求在5分钟内有效

3. **身份认证**
   - App Key验证请求来自合法应用
   - 防止未授权访问API端点

4. **完整性保护**
   - Body Hash确保数据完整性
   - HMAC-SHA256提供工业级安全

## 🔧 故障排除

### **问题1: APP无法连接API**
```
可能原因：签名中间件未正确配置
解决方案：检查API是否正确编译，查看错误日志
```

### **问题2: 401 Invalid signature
```
可能原因：密钥不匹配或签名算法错误 
解决方案：确认APP和API使用相同的SECRET_KEY
```

### **问题3: 403 Duplicate nonce
```
可能原因：网络重试导致重复请求
解决方案：APP层处理重试逻辑，或调整Nonce清理策略
```

## 🎛️ 配置选项

### **调整安全参数**

在 `record-api/src/signature.rs` 中可以调整：

- `TIMESTAMP_THRESHOLD_SECONDS`: 时间戳有效期（默认5分钟）
- Nonce清理策略：1000个Nonce后清理到500个
- `SECRET_KEY`: 签名密钥（生产环境建议通过环境变量配置）

### **生产环境建议**

```bash
# 通过环境变量配置密钥
export API_SECRET_KEY="your_production_key_here"
export API_APP_KEY="your_app_key_here"
```
